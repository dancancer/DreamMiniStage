# SillyTavern Sgw parity lab

Date: 2026-05-30

## Goal

Use a fresh upstream SillyTavern deployment plus common Sgw-related plugin assets to run the same Sgw card and compare it with DreamMiniStage. The goal is not to clone SillyTavern behavior. The goal is to identify which upstream asset semantics still matter when SillyTavern is treated only as an import source.

## What was verified

### Upstream SillyTavern

SillyTavern was started on `127.0.0.1:8001`. The card import triggered three upstream behaviors that matter:

- Embedded regex scripts were detected and imported.
- Embedded character book entries were detected and imported.
- Embedded TavernHelper script was detected and accepted.

The Sgw card has one non-playable documentation opener and 10 alternate greetings. In upstream SillyTavern, switching from opening `1/11` to `2/11` produced the playable commercial-building opening.

Three turns were sent:

1. `我在商业楼门口看到丰川祥子匆匆走出来，放慢脚步，轻声问：“你还好吗？需要我帮忙吗？”`
2. `我把手里还温热的茶递过去，语气尽量自然：“刚好买多了一瓶。你要去车站吗？我可以顺路走一段。”`
3. `我没有追上去，只是把茶收回袋子里，隔着几步跟在后面说：“那至少让我把这瓶茶留给你吧。你不用马上回答。”`

Important prompt details:

- `{{user}}` and `{{char}}` did not leak as unresolved macros.
- The request intentionally kept SFW status-template placeholders such as `{{当前日期,格式:X年X月X日星期X}}` and `{{角色名}}`. These are model-fill fields, not failed macro replacements.
- The user message carried a large `<status_current_variables>` payload generated by the TavernHelper/MVU stack.
- The SFW resource maps preserved concrete image filenames such as `b4f7ua.png` and `c0srr7.png`.

UI observation:

- The playable opening rendered richer embedded content than DreamMiniStage, including a media/player-style widget and collapsed blocks.
- In the sampled three-turn run, upstream did not reliably render a full polished SFW status panel either, but it did surface MVU-driven/collapsible sections such as `变量更新`, `小总结`, timeline, and affinity overview.
- The upstream run also exposed a weakness of the asset/preset: some assistant content leaked planning or prompt-like text. That is useful evidence for import-time QA, not something DreamMiniStage should preserve blindly.

Positive signals:

- The actual wire request now honors the streaming toggle: all three Dream requests had `stream:false`.
- The actual wire request uses `max_tokens:8192`.
- `{{user}}`, `{{char}}`, and other moustache macros did not remain unresolved.
- The prompt includes the selected playable opening, Sgw character context, world context, and a status-render contract.
- Multiple openings work after manual selection; opening `2/11` matches the playable commercial-building scene.

Differences and gaps:

1. Dream splits the compiled prompt into many small `system` messages and includes the current user instruction twice: once as a preset-style `玩家输入指令...秋青子，基于我的输入...` message and once as the final user message. Upstream sends a much more compact message list. This is not automatically wrong, but it is a source of prompt drift and should be normalized in the compiler.

2. Dream currently has an empty `<status_current_variables>` block. Upstream fills this with the card's MVU state. Without this state, affinity/status logic loses its factual input.

3. Dream normalizes the SFW template too aggressively. Upstream keeps model-fill placeholders and concrete image maps; Dream's prompt converted fields like `date`, `time`, `name`, `avatar`, and `portrait` to empty strings, and image resources degraded to `.png`. This avoids unresolved moustache text, but it removes useful schema guidance and asset linkage.

4. Dream's three assistant responses did not emit `<SFW>...</SFW>`, so no status panel was rendered. The page only showed narrative content and the old `概况 <23>` opening fragment.

5. Dream does not execute the embedded TavernHelper/MVU script. That is correct for the hard-replace direction, but the import compiler still needs to extract the deterministic asset semantics that the script was carrying: initial variables, update events, status variables, and UI-facing state transitions.

6. The Dream chat UI does not render the upstream opening's media/player widget. This should not be copied as raw HTML, but it should become an explicit unsupported/render-intent diagnostic so users know which card affordance was dropped.

7. The OpenAI-compatible direct model request is currently reachable from the browser bundle. The request-capture proxy initially failed on CORS, which proves this path is client-side. Code evidence: `lib/nodeflow/LLMNode/openai-compatible-direct.ts` performs `fetch(chatCompletionsUrl(config))` and attaches the API key in the request headers. This should move behind a server-side model gateway before product hardening.

8. The Story Agent path still logs legacy content parsing and `RegexProcessor` work at render time: `[ContentParser][ASYNC] RegexProcessor 完成，应用脚本: (无)`. That is a runtime cleanliness issue for the hard-replace route even when no script is applied.

## Interpretation

The main missing piece is not "copy SillyTavern UI". The missing piece is that Sgw's runtime behavior is partly encoded outside plain character/worldbook text:

- The opening selector matters because the first message is documentation, not a playable scene.
- MVU/TavernHelper establishes variables and state update conventions that later prompt/status logic expects.
- The status UI depends on a structured output contract plus asset maps.
- Regex/status rendering is a UI contract, not just string replacement.

DreamMiniStage already proves the most important transport fixes:

- non-streaming is honored at the actual request layer;
- `8192` output tokens are honored;
- `deepseek-v4-pro` can be driven with 1M context settings;
- imported prompt/world/character material reaches the request.

The next improvements should happen in the import compiler and runtime contract, not by embedding arbitrary ST/MVU JavaScript execution.

## Recommended next work

1. Default opener policy

   Detect documentation/non-playable first messages and default to the first playable alternate greeting, while still letting users switch. For Sgw this means defaulting to opening `2/11`.

2. MVU extraction POC

   Compile embedded TavernHelper/MVU assets into a deterministic state module:

   - initial variable schema and values;
   - update-command grammar needed by `<UpdateVariable>`;
   - current variable injection for `<status_current_variables>`;
   - event-equivalent hooks that are safe and data-driven.

3. Status contract preservation

   Do not erase model-fill placeholders into empty strings during compilation. Convert them into typed field descriptions, and preserve image map provenance so the model can select actual imported assets.

4. RenderIntent coverage for Sgw

   Add explicit render intents for:

   - SFW/NSFW status JSON panel;
   - collapsed `概况`/summary blocks;
   - affinity overview;
   - timeline/worldline section;
   - opening media/player fallback as unsupported diagnostic or safe media intent.

5. Prompt compiler normalization

   Eliminate duplicated current-user instruction in the assembled prompt. Keep one canonical current input slot and make prompt-viewer diffing show where each compiled prompt message came from.

6. Server-side model gateway

   Move OpenAI-compatible model calls out of the browser bundle. The direct client fetch path is convenient for local testing but wrong for API-key safety and CORS-independent product behavior.

7. Runtime cleanliness gate

   Story Agent runtime should not call the legacy regex/content parser path at conversation time. If content parsing is still needed for display, it should consume compiled render rules, not live ST-shaped scripts.
